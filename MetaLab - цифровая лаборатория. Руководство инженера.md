# 1. Общее описание

## 1.1 Назначение системы

**MetaLab - цифровая лаборатория** — это инженерный инструмент для виртуальной наладки алгоритмов ПЛК (программируемых логических контроллеров) через OPC UA.

**Система предназначена для:**

- ✅ **Виртуальной наладки логики управления** — проверка работы ПЛК без подключения к реальному оборудованию
- ✅ **Автономных испытаний ПЛК** — тестирование в режиме "железо в цикле" (HIL)
- ✅ **Проверки прохождения сигналов (I/O mapping)** — контроль правильности связей между тегами
- ✅ **Проверки межблокировок** — тестирование логики блокировок и защит
- ✅ **Тестирования последовательностей пуска/остановки** — отработка временных диаграмм
- ✅ **Имитации типовых отказов на логическом уровне** — проверка реакции ПЛК на аварийные ситуации
- ✅ **Проверки реакции ПЛК на изменения входов** — тестирование устойчивости алгоритмов

**Система НЕ является:**

- ❌ Цифровым двойником (не моделирует физику процессов)
- ❌ Физической моделью объекта (нет гидравлики, теплотехники и т.д.)
- ❌ Симулятором технологического процесса
- ❌ `Real`-time детерминированной системой (джиттер до секунды допустим)

## 1.2 Принцип работы

PLC работает с входами как с реальными (через переключатель real/sim в проекте), а Python-рантайм:

1. **Читает выходы PLC** (DQ_, AQ_ — дискретные и аналоговые выходы)
2. **Вычисляет входы PLC** (DI_, AI_) на основе текущего состояния и настроенных сигналов
3. **Записывает входы обратно в PLC** по OPC UA
4. **Позволяет динамически переназначать поведение входов** по событиям сценария

## 1.3 Временная модель

- Runtime работает с фиксированным шагом dt (по умолчанию 0.1 сек)
- Джиттер и небольшие задержки **некритичны** для целевого применения
- Погрешность в секунды не влияет на логику работы
- Время используется только для:
  - Задержек (`wait`, `delay`)
  - Разгонов и остановов двигателей (`runup_time`, `cooldown_time`)
  - Аналоговых сигналов (`ramp`, `sin`)

# 2. Архитектура системы

## 2.1 Основные компоненты

```text

┌─────────────────────────────────────────────────────────────┐
│                           MetaLab                           │
├───────────────┬───────────────────────────────┬─────────────┤
│  opcua_client │        core_runtime           │  signals    │
│  • connect    │  • Context (t, tags)          │  • Sensor   │
│  • read       │  • Runtime (фоновый цикл)     │  • Engine   │
│  • write      │  • Управление сигналами       │  • Valve    │
│  • subscribe  │  • Синхронизация              │  • Delay    │
└───────────────┴───────────────────────────────┴─────────────┘
        ▲                       ▲                       ▲
        └───────────────────────┼───────────────────────┘
                                │
                    ┌───────────┴───────────┐
                    │    scenario engine    │
                    │  • загрузка YAML      │
                    │  • выполнение шагов   │
                    │  • helper лог/confirm │
                    └───────────────────────┘
```

### 2.1.1 `opcua_client.py`

OPC UA клиент-обёртка с поддержкой:

- Подключение/отключение с автоматическим переподключением
- Пакетное чтение выходов PLC (bulk-read)
- Пакетная запись входов PLC (bulk-write)
- Подписка на изменения тегов (subscription)
- Преобразование типов данных (`BOOL`, `INT`, `REAL` и т.д.)

### 2.1.2 `core_runtime.py`

Ядро исполнения:

- **Context** — хранит текущее время и последние значения выходов PLC
- **Runtime** — фоновый поток, выполняющий каждый тик:
  - Чтение выходов PLC
  - Обновление Context
  - Вычисление значений всех зарегистрированных сигналов
  - Запись входов PLC

### 2.1.3 `signals.py`

Библиотека сигналов-генераторов:

- **`SensorSignal`** — аналоговые датчики (синус, линейное изменение, константа)
- **`EngineSignal`** — симуляция двигателя (пуск/останов с временем разгона)
- **`ValveSignal`** — симуляция клапана (плавное открытие/закрытие)
- **`DelaySignal`** — задержка включения/выключения
- **`ExpressionSignal`** — логические выражения над тегами

### 2.1.4 `engine.py`

Движок интерпретации сценариев:

- Загрузка конфигурации из YAML
- Загрузка шагов сценария
- Создание сигналов
- Выполнение шагов с поддержкой helper-методов

### 2.1.5 `excel2yaml.py`

Конвертер Excel → YAML:

- Преобразует листы Excel в YAML-конфигурацию
- Автоматически создает `config.yaml` и `scenario.yaml`
- Поддерживает все типы сигналов

# 3. Быстрый старт

## 3.1 Установка

**Требования**

- Python 3.8 или выше
- Зависимости из requirements.txt
- UAExpert

**Установка зависимостей**

```bash
pip install -r requirements.txt
```

**Структура проекта**

```text
plant-simulator/
├── core/
│   ├── opcua_client.py
│   ├── core_runtime.py
│   ├── signals.py
│   ├── engine.py
│   └── constants.py
├── tools/
│   ├── excel2yaml.py
│   └── run.py
├── scenarios/          # Папка для сохраненных сценариев
├── logs/               # Папка для логов
├── run.bat             # Запуск на Windows
└── requirements.txt
```

# 4. Работа с Excel-сценариями

## 4.1 Общая структура Excel-файла

Excel-файл должен содержать следующие листы (в любом порядке):

| Лист | Обязательный | Назначение |
| --- | --- | --- |
| Tags Out | ✅ | Входы ПЛК (DI_, AI_) — то, что симулятор пишет в ПЛК |
| Tags In | ✅ | Выходы ПЛК (DQ_, AQ_) — то, что симулятор читает из ПЛК |
| Sensor | ❌ | Аналоговые датчики |
| Engine | ❌ | Двигатели |
| Valve | ❌ | Клапаны |
| Delay | ❌ | Задержки |
| Scenario | ✅ | Шаги сценария |

## 4.2 Лист "Tags Out" (входы ПЛК)

Определяет теги, которые симулятор будет **записывать в ПЛК** (DI_, AI_).

| Колонка | Описание | Пример |
| --- | --- | --- |
| Name | Имя тега (должно совпадать с проектом ПЛК) | `DI_K31_bOverheating` |
| Data Type | Тип данных: `Bool`, `Int`, `Real`, `Word`, `DInt` | `Bool` |
| Value | Начальное значение (опционально) | 1 |
| Comment | Описание (опционально) | Перегрев насоса |

**Важно:** Имена тегов должны точно соответствовать тому, что настроено в OPC UA сервере ПЛК.

**Пример:**

| Name | Data Type | Value | Comment |
| --- | --- | --- | --- |
| `DI_K31_bOverheating` | `Bool` | 1 | Перегрев насоса К31 |
| `DI_K32_bOverheating` | `Bool` | 1 | Перегрев насоса К32 |
| `AI_TE1_GVS_Supply_iT` | `Int` | 5530 | Температура ГВС |

## 4.3 Лист "Tags In" (выходы ПЛК)

Определяет теги, которые симулятор будет **читать из ПЛК** (DQ_, AQ_).

| Колонка | Описание | Пример |
| --- | --- | --- |
| Name | Имя тега | `DQ_K31_Pump_bStart` |
| Data Type | Тип данных | `Bool` |
| Comment | Описание | ПУСК насоса К31 |

**Пример:**

| Name | Data Type | Comment |
| --- | --- | --- |
| `DQ_K31_Pump_bStart` | `Bool` | ПУСК насоса К31 |
| `DQ_K32_Pump_bStart` | `Bool` | ПУСК насоса К32 |
| `AQ_M21_GvsValve_iPos` | `Int` | Управление клапаном |

## 4.4 Лист "Sensor" (аналоговые датчики)

Создает сигналы типа `SensorSignal` для генерации аналоговых значений.

| Колонка | Описание | Обязательный | Пример |
| --- | --- | --- | --- |
| Name | Имя сигнала (для обращения в сценарии) | ✅ | `TE1_GVS_Supply_T` |
| Tag | Тег входа ПЛК (из Tags Out) | ✅ | `AI_TE1_GVS_Supply_iT` |
| Value | Начальное значение | ❌ (по умолч. 0) | 5530 |
| Min | Минимум для `sin` режима | ❌ | 5530 |
| Max | Максимум для `sin` режима | ❌ | 27648 |
| Period | Период для `sin`/`ramp` режимов (сек) | ❌ (по умолч. 10) | 5 |
| Comment | Описание | ❌ | Температура ГВС |

**Пример:**

| Name | Tag | Value | Min | Max | Period | Comment |
| --- | --- | --- | --- | --- | --- | --- |
| `TE1_GVS_Supply_T` | `AI_TE1_GVS_Supply_iT` | 5530 | 5530 | 27648 | 5 | Температура ГВС |
| `PE1_Vent_Supply_P` | `AI_PE1_Vent_Supply_iP` | 5530 | 0 | 10000 | 10 | Давление |

## 4.5 Лист "Engine" (двигатели)

Создает сигналы типа `EngineSignal` для симуляции двигателей.

| Колонка | Описание | Обязательный | Пример |
| --- | --- | --- | --- |
| Name | Имя сигнала | ✅ | K31 |
| Start Tag | Тег команды пуска (из Tags In) | ✅ | `DQ_K31_Pump_bStart` |

| Local Control Tag | Тег местного управления | ❌ | `DI_K31_bLocal` |
| Remote Control Tag | Тег удаленного управления | ❌ | `DI_K31_bRemote` |

**Пример:**

| K31 | `DQ_K31_Pump_bStart` | `DI_K31_bState` | `DI_K31_bLocal` | `DI_K31_bRemote` | 2.0 | 1.0 | Насос К31 |

# 5. Сигналы (Signals) — полный справочник

## 5.1 `SensorSignal` (датчик)

Аналоговый сигнал с поддержкой различных режимов генерации.

**Конструктор (в конфигурации Excel/YAML)**

| Параметр | Тип | Описание |
| --- | --- | --- |
| name | str | Имя сигнала (для обращения в сценарии) |
| tag | str | Тег входа ПЛК (из Tags Out) |
| value | float | Начальное значение |

## 5.2 `EngineSignal` (двигатель)

Симулятор двигателя с временем разгона и останова, режимами управления и возможностью имитации отказа при пуске.

**Конструктор (в конфигурации Excel/YAML)**

| Параметр | Тип | Описание |
| --- | --- | --- |
| name | str | Имя сигнала |
| `start_tag` | str | Тег команды пуска (из Tags In) |
| `state_tag` | str | Тег состояния (в Tags Out) |
| `local_control_tag` | str | Тег местного управления |
| `remote_control_tag` | str | Тег удаленного управления |
| `runup_time` | float | Время разгона (сек) |
| `cooldown_time` | float | Время останова (сек) |
| `comment` | str | Описание |
| `debug` | bool | Режим отладки |

### Методы

**Режимы управления**

#### `mode_local() -> None`

Включает местный режим: `local_control_tag = True`, `remote_control_tag = False`

Пример:

| Object | Method | Comment |
| --- | --- | --- |
| `K31` | `mode_local` | Местный режим |

#### `mode_remote() -> None`

Включает удаленный режим: `local_control_tag = False`, `remote_control_tag = True`

#### `mode_off() -> None`

Отключает оба режима: `local_control_tag = False`, `remote_control_tag = False`

**Режимы пуска**

#### `start_normal() -> None`

Нормальный пуск — двигатель запустится после `runup_time`.

#### `start_fail() -> None`

Отказ при пуске — двигатель НЕ запустится, даже если есть команда.

**Настройка параметров**

#### `set_start_tag(tag: str) -> None`

Изменяет тег команды пуска. Можно задать логическое выражение.

Пример с выражением:

| Object | Method | Tag | Comment |
| --- | --- | --- | --- |
| `K31` | `set_start_tag` | `DQ_K31_Pump_bStart` & `DI_PowerOn` | Пуск только при питании |

#### `set_runup_time(time: float) -> None`

Изменяет время разгона.

#### `set_cooldown_time(time: float) -> None`

Изменяет время останова.

**Логика работы**


1. Двигатель получает команду из `start_tag` (или из выражения)
2. При появлении команды:
    - Начинается отсчет `runup_time`
    - Если в течение разгона команда пропадает — двигатель не запускается
    - Если `start_fail`() активен — двигатель игнорирует команду
3. После `runup_time` двигатель переходит в состояние "работает"
4. При снятии команды:
    - Начинается отсчет `cooldown_time`
    - После `cooldown_time` двигатель останавливается

**Примеры**

Простой запуск:

| Object | Method | Comment |
| --- | --- | --- |
| `K31` | `mode_local` |  |
| `K31` | `start_normal` |  |
| `h` | `wait` | 5 |
| `K31` | `mode_off` |  |

Тестирование отказа:

| Object | Method | Comment |
| --- | --- | --- |
| `K31` | `mode_local` |  |
| `K31` | `start_fail` | Будет игнорировать пуск |
| `h` | `wait` | 5 |
| `K31` | `start_normal` | Теперь запустится |

Динамическое изменение времени:

| Object | Method | Value | Comment |
| --- | --- | --- | --- |
| `K31` | `set_runup_time` | 5 | Разгон 5 сек |
| `K31` | `mode_local` |  |  |
| `K31` | `start_normal` |  |  |

## 5.3 `ValveSignal` (клапан)

Симулятор клапана с плавным изменением положения и концевиками.

**Конструктор**

| Параметр | Тип | Описание |
| --- | --- | --- |
| `name` | str | Имя сигнала |
| `open_tag` | str | Команда открытия (из Tags In) |
| `close_tag` | str | Команда закрытия (из Tags In) |
| `opened_tag` | str | Концевик открыто (в Tags Out) |
| `closed_tag` | str | Концевик закрыто (в Tags Out) |
| `local_control_tag` | str | Тег местного управления |
| `remote_control_tag` | str | Тег удаленного управления |
| `full_time` | float | Время полного хода (сек) |
| `comment` | str | Описание |
| `debug` | bool | Режим отладки |

### Методы

**Режимы управления (аналогично Engine)**

#### `mode_local() -> None`

Местный режим: `local_control_tag = True`, `remote_control_tag = False`

#### `mode_remote() -> None`

Удаленный режим: `local_control_tag = False`, `remote_control_tag = True`

#### `mode_off() -> None`

Режимы выключены

**Режимы работы**

#### `normal() -> None`

Нормальный режим — концевики работают.

#### `fail() -> None`

Режим аварии — концевики НЕ срабатывают (положение замораживается).

**Настройка тегов**

#### `set_open_tag(tag: str) -> None`

Изменяет команду открытия. Поддерживает выражения.

#### `set_close_tag(tag: str) -> None`

Изменяет команду закрытия. Поддерживает выражения.

#### `set_opened_tag(tag: Optional[str]) -> None`

Изменяет тег концевика "открыто".

#### `set_closed_tag(tag: Optional[str]) -> None`

Изменяет тег концевика "закрыто".

#### `set_local_control_tag(tag: Optional[str]) -> None`

Изменяет тег местного управления.

#### `set_remote_control_tag(tag: Optional[str]) -> None`

Изменяет тег удаленного управления.

**Настройка параметров**

#### `set_full_time(time: float) -> None`

Изменяет время полного хода.

**Дополнительные методы**

`get_position() -> float`
Возвращает текущее положение клапана (0-100%).

#### `set_position(position: float) -> None`

Принудительно устанавливает положение клапана.

**Логика работы**


1. Клапан получает команды из `open_tag` и `close_tag`
2. Если `open_tag = True` и `close_tag = False` — клапан открывается
3. Если `open_tag = False` и `close_tag = True` — клапан закрывается
4. Если обе команды одинаковы — клапан стоит
5. Положение изменяется со скоростью 100% / `full_time`
6. Концевики срабатывают при достижении 0% или 100%
7. В режиме `fail`() положение замораживается, концевики = False

**Примеры**

Простое управление:

| Object | Method | Comment |
| --- | --- | --- |
| `YA1` | `mode_local` |  |
| `YA1` | `normal` |  |
| `h` | `confirm` | Открыть клапан? |
| `YA1` | `set_open_tag` | 1 (в PLC подается команда) |
| `h` | `wait` | 5 |
| `h` | `confirm` | Закрыть клапан? |
| `YA1` | `set_close_tag` | 1 |

Тестирование аварии:

| Object | Method | Comment |
| --- | --- | --- |
| `YA1` | `fail` | Аварийный режим |
| `h` | `confirm` | Подать команду открытия |
| `h` | `wait` | 10 |
| `h` | `log` | Концевики не сработают |

Принудительная установка:

| Object | Method | Value | Comment |
| --- | --- | --- | --- |
| `YA1` | `set_position` | 50 | Установить 50% |
| `h` | `confirm` | Проверить положение |  |

## 5.5 `ExpressionSignal` (логическое выражение)

Сигнал для вычисления логических выражений над тегами.

**Важно:** `ExpressionSignal` создается автоматически, если в `start_tag` или open_tag используется выражение. Обычно не требуется создавать его вручную.

**Поддерживаемые операторы**
- & — логическое И
- | — логическое ИЛИ
- ! — логическое НЕ
- ( и ) — группировка

**Примеры выражений**

| Выражение | Описание |
| --- | --- |
| `DQ_K31_Pump_bStart` | Простой тег |
| !`DQ_K31_Pump_bStart` | Инверсия |
| `DQ_K31_Pump_bStart` & `DI_PowerOn` | И |
| `DQ_K31_Pump_bStart` \| `DQ_K32_Pump_bStart` | ИЛИ |
| (`DQ_K31_Pump_bStart` \| `DQ_K32_Pump_bStart`) & `DI_PowerOn` | Группировка |

**Использование в сценарии**

```yaml

# В YAML конфигурации сигнала:
start_tag: "(DQ_K31_Pump_bStart | DQ_K32_Pump_bStart) & DI_PowerOn"
```

В Excel:
| Object | Method | Tag | Comment |
|--------|--------|-----|---------|
| `K31` | `set_start_tag` | (`DQ_K31_Pump_bStart` | `DQ_K32_Pump_bStart`) & `DI_PowerOn` |

# 10. Настройка OPC UA сервера в TIA Portal

## 10.1 Необходимое ПО

- Siemens TIA Portal (V15.1 или выше)
- Siemens PLCSIM Advanced (поддерживает OPC UA в симуляции)

## 10.2 Настройка сетевого адаптера

После установки PLCSIM Advanced в системе появится виртуальный адаптер **Siemens PLCSIM Virtual Ethernet Adapter**.

1. Откройте **Панель управления → Сетевые подключения**
2. Выберите адаптер → **Свойства → IP версии 4 (TCP/IPv4)**
3. Задайте IP-адрес из той же подсети, что и у ПЛК:
   - ПЛК: `192.168.0.1`
   - Адаптер: `192.168.0.100`
4. Маска подсети: `255.255.255.0`

## 10.3 Настройка проекта в TIA Portal

### IP-адрес ПЛК

1. В конфигурации устройства выберите ПЛК
2. В инспекторе свойств откройте раздел **Ethernet-адреса**
3. Задайте IP: `192.168.0.1`

### Включение OPC UA сервера

1. Выберите ПЛК → **Properties → OPC UA → Server**
2. Отметьте **Activate OPC UA Server**
3. На вкладке **General** запомните адрес из поля *Server addresses*

### Настройка лицензии

1. Выберите ПЛК → **Properties → Runtime Licenses**
2. В разделе **OPC UA** выберите соответствующую лицензию

### Настройка прав доступа

Отключить **CPU** -> **Properties** -> **Protection & Security** -> **Access contro**l -> **Disable access control**

**или**

1. Откройте **Project tree → Security settings → Users and roles**
2. Перейдите на вкладку **Roles** → нажмите *Add new role*
3. На вкладке **Runtime rights** активируйте:
   - Full access
   - OPC UA server access
4. Перейдите на вкладку **Users**
5. Выберите **Anonymous** → активируйте созданную роль

### Включение поддержки симуляции

1. В Project tree нажмите ПКМ по ПРОЕКТУ → **Properties**
2. Откройте вкладку **Protection**
3. Активируйте:
   - Support simulation during block compilation

## 10.4 Настройка PLCSIM Advanced

1. Запустите **Siemens PLCSIM Advanced**
2. В окне настроек укажите:
   - Online Access: `TCP/IP Single Adapter`
   - TCP/IP Communication with: `Ethernet`
   - Instance name: `MySimulation`
   - IP address / Subnet mask: `192.168.0.1 / 255.255.255.0`
3. Нажмите **Start**

## 10.5 Загрузка проекта

1. В TIA Portal выберите проект → **Download to device**
2. В поле PG/PC interface выберите **PLCSIM Virtual Ethernet Adapter**
3. Загрузите проект

## 10.6 Проверка подключения (UaExpert)

1. Запустите **UaExpert**
2. Откройте **Project → Servers → ПКМ → Add**
3. На вкладке **Advanced** вставьте endpoint:
   - `opc.tcp://192.168.0.1:4840`
4. Нажмите **OK**
5. При первом подключении подтвердите сертификат
